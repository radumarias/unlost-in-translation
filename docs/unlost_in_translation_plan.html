Unlost In Translation

Ensuring message meaning is preserved across languages and cultures. An architecture and roadmap document.

1. The Core Idea & Value Proposition

Traditional translation tools (like Google Translate) map words to words. They often fail when context changes (e.g., "bank" of a river vs. financial "bank"), when idioms are used ("break a leg"), or when tone dictates vocabulary (formal vs. informal). Unlost In Translation solves this by shifting from literal translation to intent and semantic translation.

Key Differentiators:

  • The Sanity Check Badge: Before the translation is delivered to the receiver, the sender gets a short back-translation representing how the message will be perceived in the target language. This provides immediate confidence.
  • Context Sliding Window: The app maintains a running history (e.g., the last 5-10 messages). The LLM uses this to disambiguate words and maintain a consistent tone (e.g., remembering that the conversation is about fishing, not finance).
  • Cultural & Offense Flags: If an expression translates literally into something offensive or highly unnatural in the target culture, the AI blocks or flags the translation and suggests a localized alternative.
  • Shared AI Room (3-Party Architecture): The conversation is treated as a room with three participants: User A, User B, and the AI Interpreter. This allows for seamless transitions from sharing one screen to connecting remotely via a link.

2. Detailed User Experience (UX) Flow

The UX is designed to minimize friction while maximizing trust. Here is the step-by-step flow for an in-person interaction:

  1. Initialization: User A selects their language (English) and User B's language (Spanish). The app establishes an empty context window.
  2. Input: User A taps the microphone and speaks: "This food is the bomb!" (Idiom for 'delicious').
  3. Processing:
    • The device records audio and sends it to the STT (Speech-to-Text) backend.
    • The backend receives "This food is the bomb!"
    • The LLM translates it contextually. It realizes "the bomb" in a restaurant context means excellent, not an explosive.
  4. Sanity Check & Output Generation:
    • Target Translation: "¡Esta comida es increíble!"
    • Back-translation (Intent): "This food is incredible!"
    • Cultural Flag: None (it successfully bypassed a literal translation of 'bomba').
  5. Review & Delivery:
    UI displays to User A:
    Original: "This food is the bomb!"
    Sanity Check: Translated as: "This food is incredible." (Safe)

    If Auto-Send is off, User A taps "Approve". The device rotates the text "¡Esta comida es increíble!" toward User B and plays the TTS (Text-to-Speech) audio in Spanish.

3. Technical Discussions & Stack Decisions

A deep dive into why specific technologies were chosen over alternatives to achieve low latency, high quality, and multi-device support.

Frontend Frameworks

  • Kotlin Multiplatform (KMP): Considered for native Android/iOS performance. Rejected for now because the ecosystem for cross-platform audio streaming and custom UI rendering is less mature than Flutter, requiring more boilerplate Swift code for iOS.
  • Flutter: Selected for Phase 2. Offers excellent "Hot Reload" for rapid UI iteration, paints identical custom UIs on both OSs, and has highly mature, battle-tested packages for Audio Recording (e.g., record), playing (e.g., just_audio), and WebSocket management.
  • React / Next.js (Web): Selected for Phase 1 (Validation) and Phase 3 (Remote Mode). A Progressive Web App (PWA) allows instantaneous distribution for testing. Later, it becomes the backbone for the "connect via link" remote feature.

Audio Pipeline (STT/TTS)

  • Native Web Speech APIs: Rejected. They are free, but completely unreliable across different browsers (especially Apple devices). Quality is poor for accents.
  • Cloud Pipeline (Chosen Approach): The device is a "dumb microphone/speaker". Audio is streamed to a backend.
    • STT: OpenAI Whisper API (or Google Cloud Speech). Extremely resilient to background noise and accents.
    • TTS: OpenAI TTS or ElevenLabs for hyper-realistic voices that can convey emotion, avoiding the "robotic" feel of standard OS voices.

Backend Hosting & Serverless vs. WebSockets (Vercel)

For the backend, we considered Vercel because it supports both Node.js (Next.js native) and Python (via serverless functions in the api/ directory). However, there is a critical architectural consideration:

  • The Serverless Limitation: Vercel functions are stateless and spin down immediately after responding. Therefore, Vercel does not support persistent WebSockets natively.
  • Phase 1 Solution: We will use standard HTTP POST requests on Vercel. The phone records audio, sends the file to Vercel, waits for processing, and receives the response. This is incredibly fast to build and perfect for validating the idea.
  • Phase 2 Solution: When we transition to live audio streaming for lower latency, we will keep the AI logic on Vercel but route the audio streams through a third-party real-time service (like LiveKit, Supabase Realtime, or Pusher) that handles the persistent WebSocket connections.

4. Phased Implementation Plan

Architecture Overview

[Client: Web/Mobile/Watch] | (Audio Blob over HTTPS / WebSocket) v [Backend: Vercel Serverless (Next.js or Python)] |-- 1. STT Service (Whisper: Audio -> Text) |-- 2. Session Store (Redis/Memory: Fetches last 5 messages) |-- 3. LLM Service (Gemini/OpenAI: Text + Context -> Translation + Sanity Check) |-- 4. TTS Service (ElevenLabs: Translation Text -> Audio Blob) v [Client: Web/Mobile/Watch] (Receives JSON data + Audio Blob)

Phase 1: Validation Prototype (Fastest Iteration)

Goal: Test the core hypothesis—does the Sanity Check badge actually build trust and catch errors?

  • Stack: Next.js (React) Web App deployed on Vercel. Python FastAPI or Next.js API Routes for the backend.
  • Networking: Standard HTTP POST requests (upload audio file, await response) since Vercel does not support WebSockets.
  • Features:
    • Push-to-talk button using browser MediaRecorder API.
    • Basic split-screen CSS layout.
    • Hardcoded LLM prompts to enforce the output JSON format: { translation: "", intent: "", warning: null }.
  • Testing: Can be loaded instantly on any phone via URL. No app store reviews required.

Phase 2: The Production Mobile App

Goal: A polished, native-feeling app for daily in-person use with real-time speed.

  • Stack: Flutter for iOS and Android.
  • Networking upgrade: Integrate a real-time service (e.g., LiveKit or Supabase) alongside Vercel to support low-latency WebSockets for live audio streaming.
  • Features:
    • Complex UI animations (screen flipping, pulsing microphones).
    • Background audio playback, volume normalization.
    • Offline fallbacks (using on-device translation models for basic words if internet drops).

Phase 3: Ecosystem Expansion (Wearables & Remote)

Goal: True ubiquitous communication.

  • Remote Link Sharing: User A generates a session link (e.g., unlost.app/s/8f2a) from the Flutter app. User B opens it in their browser (the Next.js app from Phase 1) to join the 3-party AI room remotely.
  • Apple Watch / WearOS: Build a lightweight native WatchOS app (SwiftUI/Kotlin). Since the backend does all the work, the watch only needs to record audio, send it via HTTP/Sockets, display text, and play the response.
